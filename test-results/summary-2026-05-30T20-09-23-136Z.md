# Chains Project Review

## 1. What the Project Does

**Chains** is a browser-based molecule-placement puzzle game with roguelike elements, inspired by 1990s atom-link arcade puzzlers. Players place atoms (Spark, Ember, Vine, Nova, Flux) from a queue onto a 12×12 grid, forming molecules where each atom satisfies its exact valence requirement. Completed molecules clear and earn score. The game features:

- **Core puzzle mechanics**: Bond atoms orthogonally; complete molecules (all atoms have exact bonds needed) to clear
- **Swap system**: Click occupied cells to swap with incoming atom (costs a move + 1 of 3 round swaps)
- **Roguelike progression**: Shop phases between rounds, powerup cards (standard + consumable), research grants, currency (Rachirals)
- **Boss rounds** (Round 3 of each level): 24 unique boss mechanics across 4 acts (Mirror Warden, Phage Bloom, Void Collector, Resonance Clerk, Lattice Keeper, Debt Engine, Entropy Saint, Final Exam, etc.)
- **Accessibility**: 4 color blindness modes, 3 bond color schemes, combo meter, onboarding tutorial

## 4. Potential Issues or Improvements

### Critical

1. **No PWA service worker**: `manifest.webmanifest` exists but no `sw.js` for offline capability or caching.

2. **Render exceptions not gracefully handled**: If `drawBoard()`, `updateLabels()`, or any visual method throws, the game hangs with no recovery.

### High Priority

3. **Extract rendering modules**: main.js handles too much — split into `src/renderer/`, `src/effects/`, `src/shop-renderer.js`, `src/ui-widgets.js`.

4. **Add type checking**: At minimum, JSDoc type annotations. Consider TypeScript for state shape, card definitions, and API contracts.

5. **Bundle for production**: No minification, no tree-shaking. Even a simple Rollup step would reduce payload size.

### Medium Priority

6. **Memory optimization for undo**: Consider incremental state diffs instead of full snapshots, or cap history length.

7. **Error boundary component**: Wrap `<canvas>` and UI in a try/catch with fallback rendering + toast message.

8. **Config file for magic numbers**: Extract constants like `SHOP_REROLL_COST`, `BASE_SWAPS_PER_ROUND`, combo thresholds to a `config.js`.

### Low Priority

9. **Asset preloading**: Card SVGs use `loading="lazy"` — consider a preload strategy for first-visit performance.

10. **Mobile gesture support**: Pinch-to-zoom, long-press for swap preview, haptic feedback.

11. **Analytics hook**: No telemetry for balance tuning. A lightweight event system (`onMoleculeClear`, `onBossDefeat`) would help.

---

## Summary

**Chains** is a well-crafted, polished puzzle game with a strong architecture foundation. The separation of concerns is exemplary — game logic is truly pure, content is data-driven, and rendering is decoupled. The roguelike systems (shop, cards, bosses, grants) are extensive and well-documented in the `docs/` folder.

The main risk is **maintainability** as the codebase grows. At 80KB, the main.js renderer is the largest concern and should be split. The lack of TypeScript and the monolithic render loop also increase the risk of subtle bugs as new features are added.

**Recommendation**: Prioritize extracting the rendering modules and adding TypeScript annotations for state shapes. This would significantly improve long-term maintainability without changing the game's behavior.
